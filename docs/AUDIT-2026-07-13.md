# Security / Bugs / Performance audit — 2026-07-13

Full audit of SidebarMonitor across three dimensions (security, correctness, performance +
maintainability), by reading every hot path and trust boundary. Everything marked **✅ Applied** shipped
in **v1.2.4**; **📝 Noted** items were left unchanged on purpose (need signing infra, real-hardware
testing, or a bigger change) and are recorded here so the decision is deliberate.

---

## ✅ Applied in v1.2.4

### Security
- **Download-URL validation** — the updater now rejects any asset URL that isn't HTTPS on a GitHub host
  before downloading, and builds the local MSI filename from a fixed flavour constant instead of the
  release URL (removes a fragile VBS-quoting path). *(Updater.cs)*

### Correctness
- **Helper publish timer made non-reentrant** *(most important bug)* — `System.Threading.Timer` fires
  concurrently if a callback outlasts its period (a slow `Win32_Process` WMI query can exceed 1 s). Two
  callbacks were racing the single-writer seqlock and could tear the published snapshot. Now an
  `Interlocked` guard drops overlapping ticks. *(Etw/Program.cs)*
- **>16-core CPUs** (Threadripper/EPYC) — per-core temp/C0 mapping is clamped to the 16-wide inline
  array; the physical-core count is capped at publish. No out-of-bounds read. *(Etw + Agent/Program.cs)*
- **Updater robustness** — reentrancy guard (`_updateBusy`) blocks a double-apply; `_latest` is
  snapshotted before the confirm dialog so a concurrent "Check now" can't null it mid-install;
  pre-release tags (`v1.2.3-rc1`) parse; the streaming download body has a 10-min timeout (the
  `HttpClient.Timeout` only bounded headers); a glitch >6 GHz clock read no longer pins the boost peak.
  *(Updater.cs, MainWindow.Updates.cs, Etw/Program.cs)*
- **Atomic config write** — `ui.json` is written to a temp file then swapped in (`File.Replace`) with a
  `.bak` fallback on load; a crash or two writers mid-write can no longer leave truncated JSON that
  silently resets every setting. *(UiConfig.cs)*
- **CSV header** waits for a populated core count (no zero-core columns for the whole file). *(CsvLogger.cs)*
- **NIC counter wrap** clamped (a 32-bit rollover no longer shows a huge negative rate). *(Agent/Program.cs)*
- **GPU "NaN"** — detail fields render "—" for unsupported sensors. *(GpuSection.cs)*

### Performance
- **O(1) section lookup** — `Find(key)` was a LINQ scan called ~18×/tick; now a dictionary. *(MainWindow.cs)*
- **Cached pens/typefaces** — sparkline baseline/grid pens and `CoreRows` typefaces were rebuilt every
  frame; now `static readonly` frozen. *(Widgets.cs, CoreRows.cs)*

### Maintainability
- Update code split out of the 1600-line `MainWindow.cs` into `MainWindow.Updates.cs` (partial class,
  zero behaviour change); version-string formatting centralized in `Updater.Format`.

## 📝 Noted — not applied (deliberate)

### Security (needs infrastructure)
- **HIGH — unverified MSI + user-writable staging (privilege escalation).** The updater downloads the
  MSI to `%LOCALAPPDATA%\…\update\` (writable by the user) and runs it via elevated `msiexec` with **no
  Authenticode/hash verification**. A same-user, medium-integrity process (commodity malware) could
  replace the MSI between download and execute (TOCTOU) → code runs elevated/SYSTEM; and a compromised
  GitHub release would be accepted. **Real fix needs code signing** (verify the MSI's signature/subject
  before running) or a signed SHA-256 published per release — both depend on the SignPath onboarding
  that's still pending. Until then this is the one material escalation path; the URL-validation above
  narrows it but doesn't close it. **Recommend: land signature verification the moment SignPath is set
  up, and gate the silent auto-install on it.**
- **LOW/INFO — command lines disclosed.** The elevated helper reads every process's command line (via
  WMI) and republishes the per-core dominant process's parent + truncated cmdline into shared memory
  that any same-session process can read. Command lines occasionally carry secrets. Consider exposing
  only image/parent names, or labelling the map to the UI's integrity level.
- **LOW — shared-memory maps writable by same-session processes.** Single-writer is a named mutex, not a
  map ACL, so a same-user process could spoof telemetry (data spoofing only, same-user boundary). The
  elevated helper's map is protected by the integrity label (medium can't write up). Optional: restrict
  the MMF DACL. Counts read from the map are safe against overflow (string reads are length-bounded, the
  whole struct is copied by value) but an oversized spoofed count would `IndexOutOfRange` (crash/DoS, not
  memory corruption) — a defensive `Math.Min(count, Max)` at consumer loops would harden it.

### Installer (needs a real-target test)
- **Hardcoded `C:\Program Files` in the helper-task path.** `build.ps1` bakes the build machine's
  `%ProgramFiles%` into the task XML, and the `CreateHelperTask` custom action hardcodes it. On a target
  whose 64-bit Program Files isn't `C:\Program Files`, the task registration silently fails (custom
  actions `Return="ignore"`) → no elevated helper. Fix: reference `[INSTALLFOLDER]helper-task.xml`.
- **`RunHelperNow` session resolution.** The MSI registers the task with a Users-group principal +
  `HighestAvailable`; `schtasks /Run` from a deferred SYSTEM action can resolve to session 0, where the
  `Local\` shared memory is invisible to the session-1 UI, and `IgnoreNew` then blocks the logon trigger
  until reboot. The dev `install.ps1` avoids this with a per-user Interactive task. Lower confidence
  (depends on Task Scheduler behaviour); on WRCX's single-user machine the helper runs fine. Recommend
  registering the task per interactive user like `install.ps1`.

### Performance (needs testing with real data — the app is already very lean)
- PDH counter arrays reuse a native buffer per tick (alloc/free + `PtrToStructure` each call); the
  per-core performance counter is read up to 3× per tick. Buffer reuse would remove native churn.
- `FrameStats.ParseRow` splits every PresentMon CSV line (300+/s while gaming) — a `ReadOnlySpan`
  walk over the mapped columns would avoid the per-frame allocations.
- Minor per-tick `List` allocations in the CPU block; per-core `Segment` list in `CoreRows`; the
  min/max axis labels rebuild `FormattedText` every frame even when bounds are settled.

### Maintainability (deferred to keep this diff reviewable)
- Extract the rest of `MainWindow.cs` the same way the update code was: `MainWindow.Build.cs`
  (~425 lines of section-building) and `MainWindow.Placement.cs`. Pure cut/paste, safe.
- Dedupe `IntArg` (identical in all three entry points → `Shared`); name the thermal-margin magic
  numbers (12/4 °C) shared by the two `TempLevel` methods; remove confirmed-dead members
  (`ProcessNames.OnStop`, `ServiceMap.Services`, `RamInfo.Loaded`).

## PawnIO readiness (for the future Intel/AMD-advanced sensors)

The recommended seam is **not** an `ISensorSource` interface (over-engineering with one real source).
Mirror the existing `RyzenSdk` shape: a `PawnIoCpu` class with `TryOpen(out err)/TryRead(out data)/
Dispose`, a `ConsentMarker.AmdAdvancedEnabled` gate polled each window, and **one override block right
after the SDK block** in the helper's publish timer (later source overrides only the fields it owns —
Tctl over die-average, the global boost ceiling into `CpuInfo.LimitMhz`). One contract field
(`CpuLimitMhz`) + an `EtwLayout.Version` bump. `LimitMhz` is already reserved and now commented for it.
See `docs/intel-pawnio.md` and `docs/amd-advanced-pawnio.md`.
